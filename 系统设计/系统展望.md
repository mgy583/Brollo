Rust 高并发记账系统全景
1. 选题价值与创新
1.1 移动支付时代的记账痛点
场景描述：多账户、多币种、实时行情下，传统 Excel/单机 App 难整合，出现数据割裂、统计滞后、决策盲区。
用户痛点：手工对账耗时、跨平台余额分散、预算超支无预警。
结论：需一体化系统解决分散数据与实时洞察需求。
1.2 Rust 技术栈的空白突破
Node/Java 方案并发高但延迟大，内存泄漏风险高。
Rust 内存安全+异步性能优势显著，高校毕设记账场景尚无完整工程案例。
选题定位：国内首次将 Rust+Axum 引入个人记账，追求生产级并发指标。
2. 研究目标与交付
2.1 量化指标与成果形态
性能目标：单实例 5 万并发平均延迟 <60ms，分钟级资产净值更新。
交付物：可部署系统、开源代码仓库、学术论文。
验证方式：基准压测、真实场景 A/B 测试、论文重复率 ≤20%。
3. 文献与现状洞察
3.1 国内外研究空白对比
国外重安全隐私，缺高并发实时行情研究。
国内多 Node 技术栈，无 Rust 完整工程案例。
本项目填补高校毕设在高性能记账领域的空白。
4. 需求与架构设计
4.1 问卷提炼五大核心需求
多账户余额快照、批量导入智能去重、预算动态预警、多维度可视化、多端实时同步。
数据来源：312 份问卷+6 款竞品比对，需求真实可信。
4.2 微服务架构与数据模型
技术栈：Rust+Axum，MongoDB 事务数据，Redis 热点缓存，React+Vite 前端。
数据模型：用户、账户、分类、预算、交易五集合，复合索引保证分页 <50ms。
5. 算法与功能模块
5.1 滑动窗口预算预测算法
30 天滑动窗口+加权移动平均+节假日因子修正，实现分钟级预算偏差预警。
时间复杂度 O(n)，支持实时流式更新。
5.2 多源行情融合与模块职责
卡尔曼滤波融合 Yahoo Finance、央行汇率、手动估值，输出统一资产净值。
功能模块：账户管理、收支录入、预算预警、报表导出、仪表盘定制、系统管理六子系统，覆盖个人财务全生命周期。
6. 研究方法与路线
6.1 敏捷迭代与实验对比策略
方法：文献调研、软件工程、实验对比。
过程：4 轮敏捷 Sprint，每两周评审；wrk 压测 Rust vs Node，采集延迟与吞吐量。
目标：确保结果可重复、可扩展。
7. 进度计划与验证
7.1 十八周甘特里程碑
1-2 周：需求+开题；3-8 周：后端 API 完成 80%；9-12 周：前端 LCP<2.5s；13-14 周：压测 5 万并发<60ms；15 周：用户试用满意度 ≥85%；16-17 周：论文+预答辩；18 周：正式答辩。
7.2 性能指标与满意度验收
并发 5 万延迟 <60ms、分钟级净值更新、连续 30 天无故障、用户满意度 ≥85%、论文重复率 ≤20%。
逐条对照研究目标，形成需求-验证闭环。
8. 总结与展望
8.1 创新总结与未来演进
三大创新：Rust+Axum 记账框架、滑动窗口预算算法、可拖拽自定义仪表盘。
展望：开放插件市场、多云容灾、AI 自动调参，实现从个人记账到家庭财富管理平台升级。
